# 短篇小说生成技能架构优化说明

## 优化内容

### 1. 输出目录结构优化

**原结构**：
```
outputs/
├── 小说1.md
├── 小说1_meta.json
├── 小说2.md
└── 小说2_meta.json
```

**优化后结构**：
```
outputs/
├── novel-used-characters.json  # 全局角色跟踪文件
├── 小说1/
│   ├── 小说1.md              # 完整小说
│   ├── 小说1_meta.json       # 元数据
│   ├── chapters/             # 章节分文件
│   │   ├── 小说1_chapter_01.md
│   │   ├── 小说1_chapter_02.md
│   │   └── ...
│   └── work/                # 工作数据
│       ├── outline.json       # 大纲数据
       ├── characters.json    # 角色设定
       └── generation_log.json # 生成日志
└── 小说2/
    └── ...
```

### 2. 角色名称去重机制

**文件**：`outputs/novel-used-characters.json`

**功能**：
- 记录所有已使用的角色名称
- 确保每本小说的主角组合唯一
- 防止同一角色在不同小说中重复使用

**数据结构**：
```json
{
  "last_updated": "2026-03-20T10:30:00Z",
  "used_protagonist_pairs": [
    {
      "female": "林婉",
      "male": "陆沉",
      "novel_title": "ICU 门外，他抱白月光笑时，我重生了",
      "timestamp": "2026-03-18T19:30:00Z"
    }
  ],
  "used_character_names": ["林婉", "陆沉", "苏月", "陈风"],
  "available_female_names": [...],
  "available_male_names": [...],
  "available_secondary_names": [...]
}
```

### 3. 角色关系明确化

**原问题**：
- 角色关系模糊，同一角色在不同章节中身份混乱
- 出现"一下是哥哥，一下是弟弟"的情况

**优化方案**：
- 在 `novel-step-01-topic-selector` 输出中明确角色定义
- 增加角色对齐（alignment）属性：protagonist/antagonist/neutral/ally
- 增加角色关系（relationship_to_protagonist）属性
- 所有角色数据在技能间完整传递

**角色定义示例**：
```json
{
  "characters": {
    "protagonist": {
      "name": "江念",
      "gender": "女",
      "role": "女主（重生/觉醒）",
      "alignment": "protagonist",
      "personality": {
        "start": "深情付出",
        "middle": "心死觉醒",
        "end": "独立女王"
      }
    },
    "antagonist_main": {
      "name": "顾辞",
      "gender": "男",
      "role": "渣男前夫",
      "alignment": "antagonist",
      "personality": {
        "start": "冷漠轻视",
        "middle": "幡然醒悟",
        "end": "后悔追妻"
      }
    }
  }
}
```

### 4. 正反派固定机制

**问题**：同一作者下面的人一下正派、一下反派

**解决方案**：
1. 在选题阶段明确角色对齐属性
2. 将角色设定传递给所有后续技能
3. 写作时严格引用角色定义
4. 确保角色身份贯穿始终

**角色对齐（alignment）类型**：
- `protagonist`：主角（正派）
- `antagonist`：反派
- `neutral`：中立角色
- `ally`：主角盟友

### 5. 角色引用机制

**优化内容**：
- `novel-step-01-topic-selector` 生成完整角色定义
- `novel-step-02-outline-designer` 使用角色定义设计大纲
- `novel-sub-skill-novelist-author` 根据角色定义写作
- 所有技能通过 `characters` 字段接收角色数据

**角色数据传递链**：
```
topic-selector
  ↓ 生成角色定义
outline-designer
  ↓ 使用角色设计大纲
chapter-writer
  ↓ 传递角色定义
novelist-author
  ↓ 根据角色定义写作
```

## 使用说明

### 初始化角色跟踪文件

首次使用时，`outputs/novel-used-characters.json` 已创建并包含示例数据。

### 生成新小说

调用 `/novel-generator-main` 时：

1. 系统自动读取 `novel-used-characters.json`
2. 从可用名字池中选择未使用的角色名
3. 生成角色定义和小说大纲
4. 创建小说专属文件夹
5. 输出章节分文件和元数据
6. 更新角色跟踪文件

### 检查角色使用情况

查看 `outputs/novel-used-characters.json` 了解：
- 已使用的角色组合
- 剩余可用角色名
- 使用历史和小说标题

## 注意事项

1. **角色唯一性**：每本小说的主角组合必须唯一
2. **角色一致性**：角色定义在所有章节中保持一致
3. **关系清晰**：通过 `relationship_to_protagonist` 明确角色关系
4. **正反派固定**：通过 `alignment` 属性固定角色阵营
5. **数据完整传递**：角色定义在技能间完整传递

## 后续优化建议

1. **增加角色验证**：验证章节中角色名是否与定义一致
2. **增加关系检查**：检查角色关系是否符合设定
3. **增加冲突验证**：验证正反派冲突是否合理
4. **增加角色弧光追踪**：追踪角色性格发展是否符合设定
